六大购物场景.

从基础满减到预售膨胀，覆盖大促期间常见玩法

🎯

补差凑单

Gap-filling

差 ¥11 到满减门槛？算出凑单品的实际成本——有时凑单反而倒赚，帮你判断值不值得加购。

核心场景
✂️

分单优化

Order splitting

同一平台 N 件商品，自动穷举所有拆分组合（≤10 件）或贪心优化（>10 件），找到满减次数最多的分单方式。

核心场景
🏪

跨店满减

Cross-store stacking

商品分布在不同店铺？先组内优化店铺券，剩余孤品丢入跨店满减池，两层折扣叠加最大化。

核心场景
💰

定金膨胀

Deposit expansion

预售商品 ¥50 定金抵 ¥100？自动计算膨胀后的真实优惠，支持与跨店满减叠加计算。

新增
🏷️

品类券

Category coupons

平台发放的限品类优惠券，识别平行叠加 vs 递进叠加规则，避免券面额缩减的坑。

新增
↩️

凑单退款

Filler refund

按比例计算实际退款金额（非原价），并主动提示平台标记风险和信用影响。仅用户问及时解释。

新增

🔴 京东
🟡 拼多多
⚫ 抖音商城

工作流程.

四步完成：收集 → 查规则 → 算方案 → 给结果

1

收集信息

用户输入商品清单（名称、价格、数量、平台、店铺）。支持自然语言输入或结构化表单添加。
特殊身份（88VIP / PLUS）和已有优惠券一并收集。

2

确认规则

优先问用户：「看一眼 App 活动页，3 秒搞定」——比联网搜快 10 倍且绝对准确。
用户不知道时才联网搜索。搜不到自动启用兜底规则（每满300减50，连续四年未变），不阻塞流程。

3

计算方案

根据场景调用对应算法：补差凑单用模运算、分单优化用穷举/贪心、跨店满减分组后池化。
商品 ≤10 件穷举所有 2-3 单分组，>10 件贪心 + 局部调优。定金膨胀 / 品类券独立叠加。

4

输出结果

输出 2-3 个方案供对比，每个方案包含：订单明细、分项折扣（满减/品类券/店铺券/会员）、实付金额。
同时输出结构化 JSON 供 Web UI 渲染卡片式展示。

核心算法.

轻量高效的计算逻辑，前端 JS 即可运行

补差凑单 · 分单优化 · 跨店满减

// 补差凑单 target = M - (total % M) // 距下一门槛还差多少 cost = target - D // 实际多花的钱（负数=倒赚） // 分单优化（≤10 件穷举，>10 件贪心） for mask = 1 .. 2^n-2: // 枚举所有二分 saving = Σ floor(group_total / M) × D → 保留 saving 最大的分组方案 // 跨店满减 group items by store // 按店铺分组 optimize each group (分单优化) // 组内用店铺券 → 剩余孤品 → 跨店满减池统一计算

设计原则.

务实、可靠、不给用户添麻烦

🚫

不爬取实时价格

所有价格由用户手动提供。避免爬虫被封、数据不准、法律风险。用户比爬虫更清楚自己要买什么。

📱

优先问用户规则

用户的手机 App 是规则最权威的来源——比联网搜快 10 倍且绝对准确。搜索只是备选方案。

🛡️

兜底规则足够可靠

618「每满 300 减 50」连续四年未变，是行业基础设施级稳定性。搜不到就用兜底，绝不阻塞流程。

📊

多方案对比

输出 2-3 个方案而非单一答案。用户有最终决策权——你提供数据，用户做判断。

⚡

离线可用

前端内置完整计算引擎，断网也能用。API 是增强，不是依赖。本地计算即时响应，无需等待。

⚠️

主动提示风险

凑单退款等灰色操作只在用户问及时解释，且必须告知平台标记风险。不主动推荐有副作用的策略。
