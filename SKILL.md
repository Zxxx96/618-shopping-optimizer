# 618 购物凑单优化

## 使用场景

用户想在大促期间（618 / 双11 / 双十二）买东西，需要：
- 把多件商品拆分成最优订单组合以最大化利用满减
- 已知购物清单但差一点到满减门槛，需要推荐凑单品
- 跨店满减场景下计算最优店铺组合

## 核心原则

1. 永远让用户手动输入价格 —— 不爬取电商平台实时价格
2. 平台规则优先联网搜索 —— 用户只需告知平台名，agent 负责搜规则
3. 输出 2-3 个方案供用户选择，而非单一答案

## 工作流程

### Step 1: 收集信息

向用户确认以下信息（如果用户已经提供则跳过）：

- 商品清单：名称、单价、数量、所在平台（京东/淘宝/拼多多/抖音）、店铺名（跨店场景需要）
- 平台规则：**优先问用户** —— 先问用户是否知道规则（「看一眼手机 App 活动页，3 秒」）。用户不知道或坚持要你搜时，才启动联网搜索。搜索失败则直接用兜底规则，不阻塞流程
- 预算上限（可选）
- 特殊身份：Plus 会员、88VIP、省钱月卡等（影响额外折扣）
- 已有优惠券：面额和使用条件

### Step 2: 获取平台规则（双模式）

**模式 A：用户直接告知（首选，最快最准）**

主动询问用户：「你知道当前平台的 618 规则吗？看一眼手机 App 活动页念给我就行，3 秒搞定。」

用户只需说：满减门槛（如每满 300-50）、跨店规则、会员折扣。不需要精确文案。

这是首选路径 —— 用户的手机 App 是规则最权威的来源，比你联网搜快 10 倍且绝对准确。

**模式 B：联网搜索（备选，用户坚持时使用）**

只有当用户明确说「你帮我搜一下」或「我不知道规则」时才执行。

搜索关键词模板：
- 京东：「京东 2026 618 满减规则 每满」
- 淘宝/天猫：「淘宝 2026 618 跨店满减 每满」
- 拼多多：「拼多多 2026 618 满减规则」
- 抖音：「抖音商城 2026 618 满减活动」

需要提取的关键信息：
- 满减门槛和金额（如每满300减50）
- 是否支持跨店（跨店满减 vs 店铺券）
- 优惠券叠加规则（平行优惠 vs 递进优惠）
- 平台券 / 品类券 / 店铺券的区别
- 是否可与其他优惠叠加
- 是否有上限（如最多减多少）

**搜索失败怎么办：**

这不是 bug，是预期内的。原因有三：
1. 淘宝/天猫活动页是 JS 动态渲染的 SPA，curl 只能拿到空壳 HTML
2. 外部搜索引擎（Google/Bing/DuckDuckGo）对中国电商页面的索引覆盖有限
3. 618 活动在 5 月中旬可能还没上线（通常 5 月底预热）

如果搜索失败，直接告知用户搜索未果，然后：

→ **立即启用兜底规则（见下方）。** 淘宝/天猫/京东的 618 跨店满减已连续四年稳定在「每满 300 减 50」，这是行业基础设施级别的稳定规则。告知用户「基于往年惯例，实际以 App 活动页为准」，然后直接进入计算 —— 不要因为搜不到规则而阻塞整个流程。

**兜底规则（搜索失败时自动启用）：**

| 平台 | 满减规则 | 会员折扣 | 基于年份 |
|------|---------|---------|---------|
| 淘宝/天猫 | 每满300减50，跨店可叠，上不封顶 | 88VIP 折后 95 折 | 2025 |
| 京东 | 每满300减50，跨店可叠 | PLUS 会员额外折扣（通常 95 折） | 2025 |
| 拼多多 | 百亿补贴直降为主 | — | 2025 |
| 抖音 | 每满200减30 或 每满300减45 | — | 2025 |

> 启用兜底规则时补充一句：「这是基于 2025 年的规则，连续四年未变，但 2026 年如 5 月底前使用请以实际活动页为准，如有调整请指正。」
> JSON 输出中 `rule_year` 设为实际规则年份，`rule_source` 设为 `"fallback"`。

### Step 3: 建模与计算

#### 3.1 补差凑单（凑到门槛）

场景：用户有 N 件商品，总价 T，满减门槛为 M，满减金额为 D。
当前优惠 = floor(T / M) × D
差 = M - (T mod M)
目标：推荐一个价格接近「差」的凑单品，使总价刚好跨过下一个满减档位。

算法：
```
target_filler_price = M - (T % M)
effective_cost = target_filler_price - D  # 凑单后多减了 D，实际多花的钱
if effective_cost < 0:
    → 凑单反而省钱，强烈推荐
else:
    → 实际多花 effective_cost 元获得凑单品
```

输出：建议凑单价位区间，给出 3 个价格选项和对应的实际成本。

#### 3.2 分单优化（拆分订单）

场景：用户有 N 件商品，满减规则每满 M 减 D。每单满减次数 = floor(订单总价 / M)。

目标：将 N 件商品分配到 K 个订单中，使总折扣最大化。

算法思路：
1. 将商品按价格降序排列
2. 贪心分配：每单尽可能接近 M 的整数倍 + M-1（即刚好差 1 元到下一个门槛）
3. 保留少部分「余数」商品，尝试所有可能的重新分配

实现时使用以下方法：
- 全排列尝试所有 2-3 单的分组（商品数 ≤ 10 时）
- 商品数 > 10 时使用贪心 + 局部调优

输出：每单商品组成、总价、使用券数、实付金额、总节省。

#### 3.3 跨店满减

场景：商品分布在不同店铺，平台支持跨店满减。
每满 M 减 D 是跨店的，但店铺券只在单店生效。

计算时区分：
- 跨店满减池：所有参与跨店的商品总价，计算统一满减
- 店铺券：单店商品总价满足条件时额外减
- 最优策略：同店铺商品尽量凑一起用店铺券，跨店商品凑满减

算法：先按店铺分组，每组内做 3.2 分单优化，剩余孤品丢入跨店满减池。

#### 3.4 定金膨胀

场景：预售商品，用户支付定金 ¥P，膨胀后抵 ¥P×K（K 为膨胀倍数，如 1.5 倍、2 倍）。

计算：
```
膨胀金额 = P × K
实际优惠 = 膨胀金额 - P = P × (K - 1)
商品尾款 = 商品总价 - 膨胀金额
用户总支出 = 定金 + 尾款 = P + (商品总价 - P×K) = 商品总价 - P×(K-1)
```

注意事项：
- 膨胀倍数 K 每年、每商品不同，需从活动页确认或按去年惯例估算
- 定金通常不退（除非未付尾款），计入沉没成本
- 定金膨胀与跨店满减通常可叠加：先算膨胀抵扣，再对尾款部分参与满减
- 如果用户说不清膨胀规则，假设 K=1（即定金无膨胀），不阻塞流程

#### 3.5 品类券

场景：平台发放限品类的优惠券（如"数码券满1000减100"）。

关键问题：品类券与跨店满减的叠加方式。

常见规则：
- 大多数平台：品类券与跨店满减**平行叠加**（各算各的，互不影响）
- 少数情况：**递进叠加**（先满减再用券，券面额可能缩减）

计算方法：
1. 先算跨店满减折扣
2. 再对符合品类的商品总价判断是否满足品类券门槛
3. 品类券折扣独立计算，叠加在满减之后

处理方式：让用户确认叠加规则。如果不确定，默认按平行叠加处理（最常见）。

#### 3.6 凑单退款（仅用户主动问及时解释）

场景：用户想凑单后退掉凑单品，只保留想要的商品享受满减。

实际退款金额（以淘宝为例）：
```
退款比例 = 凑单品价格 / 订单总价
实际退款 = 凑单品价格 × (1 - 满减折扣/订单总价)
```

例如：订单总价 ¥850，满减 -¥50，凑单品 ¥55
退款 = 55 × (1 - 50/850) = 55 × 0.941 = ¥51.76（不是原价 ¥55）

**风险提示（必须告知用户）：**
- 频繁凑单退款可能被平台标记，影响账号信用
- 部分平台退款后会取消满减优惠（需确认规则）
- 不要主动推荐此策略，仅在用户明确询问时解释

### Step 4: 输出方案

对每个场景输出：

1. 方案概述（一句话）
2. 详细方案表格：

   - 订单 2：...
   - 总计节省：¥S
3. 方案对比（为什么这个方案好于其他）
4. 注意事项（如某些券不可叠加、88VIP 额外折扣是否已计入等）

## 注意事项

1. 本 skill 不爬取商品实时价格，所有价格由用户手动提供
2. 平台规则通过联网搜索获取，可能存在时效性差异
3. 计算不考虑运费影响（默认包邮或忽略运费）
4. 用户特殊身份（Plus/88VIP）的折扣通常最后计算，叠加在满减之后
5. 预售商品、定金膨胀等特殊玩法需要用户手动说明

## Pitfalls（踩坑记录）

1. **淘宝/天猫页面用 curl 抓不到规则。** 活动页面是 JS 动态渲染的 SPA 单页应用，curl 只能拿到 `<div id="root">` 的空壳。不要反复尝试 curl，直接走「问用户」或「兜底规则」路径。

2. **浏览器也可能打不开。** Hermes 的浏览器模块启动子进程时需要 `node` 在默认 PATH 中。如果 Node.js 通过 nvm 管理，子进程不加载 `.zshrc`，会报 `spawn node ENOENT`。修复方法：在 `config.yaml` 的 `env.PATH` 中追加 nvm node 路径，然后把 Playwright channel 设为 `chromium` 而非 `chrome`。

3. **搜索引擎可能超时或被墙。** Google、百度、DuckDuckGo 在不同网络环境下可用性差异大。不要在这步上花超过 30 秒 —— 超时就启用兜底规则，告知用户「搜索未果，按往年规则计算」。

4. **兜底规则足够可靠。** 618 跨店满减「每满 300-50」已连续四年未变，是行业基础设施级别的稳定性。用户即使不知道规则，按这个算也大概率正确。重点放在算方案上，不是搜规则。

5. **不要在没有商品清单和价格的情况下开始计算。** 这个 skill 不爬取价格 —— 用户必须手动提供每件商品的名称和价格。如果用户只说了品类没说具体型号和价格，先追问再算。

6. **`hermes gateway run --platform api_server` 是无效命令。** Gateway 不支持 `--platform` 参数。API 服务器通过 `config.yaml` 中的 `platforms.api_server` 配置段自动运行 —— 只要 `hermes gateway run` 启动就会带起来。不要在脚本或 serve.sh 里写这个命令。

7. **Gateway 已经在后台运行就不要重新启动。** 网关常驻进程可能已经托管了 API 服务器（检查 `hermes gateway status` 或 `curl localhost:8642/health`）。网页服务（http.server）是独立进程，可以单独重启。

8. **每年 5 月底前使用时主动验证兜底规则。** 618 活动通常 5 月底公布当年规则。如果在 5 月中旬使用本 skill，兜底规则基于去年，可能已过时。回复时必须标注 `rule_year`，并提示用户「如已公布新规则请指正」。

9. **始终在自然语言回复后追加 JSON 代码块。** Web UI 依赖这个 JSON 做结构化展示。即使用户通过纯文本对话（非 Web），也应输出 JSON——它不干扰阅读，且方便用户复制粘贴到表格。

10. **凑单退款不要主动推荐。** 只有用户明确问「凑完单能退吗」才解释，且必须告知风险（平台标记、信用影响、可能取消满减）。

## Web 版本部署（让其他人也能用）

### 手机访问

这个 skill 包含一个完整的手机网页，启动后任何人都能用浏览器打开。部署步骤：

```bash
# 1. 启动服务（API + 网页）
bash ~/.hermes/skills/productivity/618-shopping-optimizer/serve.sh
```

这会同时启动两样东西：
- Hermes API 服务器（端口 8642）—— 处理凑单计算
- 网页服务（端口 8080）—— 手机浏览器打开的页面

### 分享给其他人

同一局域网内，其他人用手机浏览器打开 `http://<你的电脑IP>:8080` 就能用。

获取 IP：运行 `ipconfig getifaddr en0`

### 公网访问

如果想让不在局域网的人也能用，推荐用 Cloudflare Tunnel（免费）：

```bash
# 安装 cloudflared
brew install cloudflared

# 建立隧道
cloudflared tunnel --url http://localhost:8080
```

会生成一个 `https://xxx.trycloudflare.com` 的公网地址，发给别人即可。

### 网页文件位置

- 网页：`web/index.html`
- 启动脚本：`serve.sh`

### 配置说明

`config.yaml` 中 API 服务器的配置：
```yaml
platforms:
  api_server:
    enabled: true
    extra:
      host: 0.0.0.0     # 允许局域网访问
      port: 8642
      cors_origins: "*"   # 允许网页跨域调用
```

### 网页特性

- 手机优先设计，适配所有移动设备
- 流式响应，打字机效果
- 快捷按钮：一键示例凑单场景
- 设置页：可自定义 API 地址（适合部署到服务器）
- 自动重连检测

## 示例对话

用户：我想买一双鞋 ¥459 和一件外套 ¥389，都在淘宝，有 88VIP，帮我算 618 怎么买最划算

agent 操作：
1. 搜索「淘宝 2026 618 跨店满减」
2. 得到规则：每满300减50，跨店可用
3. 计算：
   - 单看：459 + 389 = 848，满减 = 2 × 50 = 100
   - 实付 = 848 - 100 = 748
   - 88VIP 95折 = 748 × 0.95 = 710.6
   - 如果拆单：鞋459单独 → 减50=409 ×0.95=388.55
     外套389 → 差11到300门槛，现在只减50=339 ×0.95=322.05
     合计 710.6
   - 凑单方案：外套389 → 凑一件¥11+的商品到400，满减不变（已经在300-599档）
     总看：459+389+11=859，满减还是100，没变
   - 最优：不拆单，直接一起下单。848刚好错过第三个300（差52），但凑单品至少需要52元才多减50，凑单品成本52-50=2元，如果找到刚好52元的实用凑单品就值。
   
4. 输出方案：不拆单方案实付710.6；凑单52元方案实付710.6+2=712.6但多一件商品。
